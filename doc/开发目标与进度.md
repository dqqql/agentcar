# 智能座舱出行规划 Agent 开发目标与进度

## 1. 当前结论

- 项目当前状态: 刚开工，文档已具备，本地环境与 Gitee 仓库已准备好，代码尚未正式启动。
- 当前目标: 先做出一个可汇报、可演示、可逐步扩展的 MVP 闭环，而不是一开始追求完整智能座舱系统。
- 当前阶段: 执行准备阶段。
- 交付策略: 每一轮任务都必须有可见效果，方便给老师演示和汇报。

## 2. MVP 范围

### 必须做

- 用户输入一段自然语言出行需求
- 系统识别日期、目的地、景点、餐饮、预算等关键信息
- 系统基于简单规则和模拟数据生成景点、酒店、餐饮推荐
- 系统生成一份按天展示的出行规划
- 页面可以展示输入结果、推荐结果和行程结果
- 用户可以做一次简单修改，比如“更紧凑一点”或“换一家酒店”

### 可延后

- 语音输入
- 多轮复杂对话
- 真实座舱接入
- 实时价格查询
- 路线优化算法
- 深度学习推荐算法

### 现在不建议做

- 微服务拆分
- 复杂中台设计
- 大规模真实数据治理
- 为了“更像 AI”而过早做复杂 Agent 编排

## 3. 开发原则

- 先跑通，再优化
- 先 mock 数据，再接外部 API
- 先单体后端，再考虑复杂部署
- 每次任务完成后都要留下一个能截图或录屏的结果
- 优先保证表格、结构化结果、接口可读性

## 7. 各任务的最小完成标准

### T0 项目初始化

- 建立后端、前端、数据、测试目录
- 创建 `README.md`
- 创建基础依赖文件
- 本文档加入仓库并作为持续更新的主文档

### T1 后端启动

- FastAPI 启动成功
- 有 `/health` 接口
- 返回统一 JSON 格式

### T2 数据样本准备

- 至少准备 1 个城市的数据
- 每类至少 10 条样本
- 字段统一，便于后续接口调用

### T3 需求解析 v1

- 支持识别日期
- 支持识别目的地
- 支持识别目标景点和目标餐饮
- 不追求高智能，先规则可用

### T4 偏好模型 v1

- 支持默认用户画像
- 支持偏好字段: 景点、酒店、餐饮、出行风格

### T5 推荐服务 v1

- 至少输出 3 个景点
- 至少输出 3 个酒店
- 至少输出每日餐饮建议
- 用户明确指定的内容必须命中

### T6 行程生成 v1

- 能按日期生成每天计划
- 能输出自然语言摘要和结构化数据

### T7 前端演示页 v1

- 可输入需求
- 可发起请求
- 可查看结果
- 优先保证信息清晰，不追求复杂视觉

### T8 修改规划 v1

- 至少支持 2 类修改指令
- 能重新返回结果

### T9 异常处理

- 日期缺失时报错
- 日期非法时报错
- 目的地无法识别时报错
- 数据为空时返回友好提示

### T10 测试与验收

- 至少覆盖解析、推荐、规划 3 类测试
- 有基础接口测试
- 有手工验收用例

### T11 部署与汇报

- 本地可演示
- 有启动说明
- 有演示顺序
- 有老师汇报时可直接展示的截图或录屏点
