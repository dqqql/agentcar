# 智能座舱出行规划 Agent 开发目标与进度

## 1. 当前结论

- 项目当前状态: 已完成项目骨架和 FastAPI 最小后端启动，`scripts/getdata` 下已补充真实地点与餐饮数据抓取脚本，项目已进入执行阶段。
- 当前目标: 先做出一个可汇报、可演示、可逐步扩展的 MVP 闭环，而不是一开始追求完整智能座舱系统。
- 当前阶段: 执行阶段，T1 已完成，T2 数据准备进入推进中。
- 交付策略: 每一轮任务都必须有可见效果，方便给老师演示和汇报。
- 协作原则: 用户当前处于边学边做阶段，因此讲解可以从简单示例开始，但最终落地代码必须满足工程级要求。
- 当前新增能力: 已具备通过高德 API 抓取地点 POI 和周边餐饮数据的脚本能力，为后续推荐数据样本沉淀提供基础。
- 当前主要缺口: 真实抓取结果和字段标准尚未正式沉淀到 `data/` 目录，因此 T2 还不能判定为完成。

## 1.1 双轨协作原则

### 解释层

- 解释尽量短，优先回答当前最核心的问题
- 允许先用单文件、最小示例帮助理解
- 优先帮助用户建立“先跑通，再理解，再抽象”的认知路径

### 落地层

- 一旦进入正式功能开发，代码必须按工程化标准落地
- 所有后续代码默认要求可维护、可扩展、可测试
- 不因为讲解简单而牺牲代码质量

## 1.2 工程级代码标准

- 模块职责清晰，避免所有逻辑堆在入口文件
- 接口输入输出结构明确，返回格式统一
- 关键逻辑具备类型标注
- 关键模块具备基础测试
- 配置、数据、业务逻辑分离
- 异常处理明确，不依赖“跑错了再看”
- 文档持续同步更新，代码与文档不能脱节
- 当最小实现不再适合继续扩展时，必须主动重构，不拖到后期

## 2. MVP 范围

### 必须做

- 用户输入一段自然语言出行需求
- 系统识别日期、目的地、景点、餐饮、预算等关键信息
- 系统基于简单规则和模拟数据生成景点、酒店、餐饮推荐
- 系统生成一份按天展示的出行规划
- 页面可以展示输入结果、推荐结果和行程结果
- 用户可以做一次简单修改，比如“更紧凑一点”或“换一家酒店”

### 可延后

- 语音输入
- 多轮复杂对话
- 真实座舱接入
- 实时价格查询
- 路线优化算法
- 深度学习推荐算法

### 现在不建议做

- 微服务拆分
- 复杂中台设计
- 大规模真实数据治理
- 为了“更像 AI”而过早做复杂 Agent 编排

## 3. 开发原则

- 先跑通，再优化
- 先 mock 数据，再接外部 API
- 先单体后端，再考虑复杂部署
- 每次任务完成后都要留下一个能截图或录屏的结果

- 每完成一个阶段，都要判断是否需要从“学习型代码”升级到“工程型代码”

## 6.1 代码升级策略

### 当前策略

- 当前用户处于学习阶段，可以从简单写法切入
- 但后续每进入一个核心阶段，都要回头做一次结构升级

### 升级要求

- `main.py` 只保留应用入口和基础装配逻辑
- 需求解析、推荐、规划必须拆到独立模块

- 测试必须覆盖解析、推荐、规划 3 个主模块
